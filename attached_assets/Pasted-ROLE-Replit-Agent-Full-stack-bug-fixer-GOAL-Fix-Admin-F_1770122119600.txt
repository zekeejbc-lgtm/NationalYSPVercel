TASK A: FIX “FAILED TO DELETE CHAPTER”
1) Locate where the error message “Failed to Delete Chapter” is triggered in the Admin UI.
2) Identify which API call is used (route + method), e.g. DELETE /api/chapters/:id.
3) Confirm the backend route exists and matches the frontend call exactly:
   - HTTP method must match (DELETE vs POST).
   - URL path must match.
   - If using fetch/axios, confirm correct headers and credentials.
4) Add robust backend logging for this endpoint:
   - Log request params (chapterId) and the DB result (rows affected / deleted document).
   - Log and return server-side error messages (without leaking secrets).
5) Common root causes to check and fix:
   - The chapterId is undefined/null in the request.
   - Endpoint expects `id` but frontend sends `chapterId` (or vice versa).
   - Soft delete vs hard delete mismatch.

   - Authorization middleware blocking delete.
6) If foreign key constraints exist:
   - Decide the safest low-cost behavior:

   - Handle non-200 responses: show backend error message.
